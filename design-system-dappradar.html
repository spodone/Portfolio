    02 — Context

    Why this mattered

    DappRadar is the world's leading Web3 data platform — tracking dApps, NFTs, DeFi, and blockchain activity across 50+ chains. When I joined as Lead Product Designer, the product had grown rapidly without a shared design foundation. Each vertical (Rankings, Portfolio, Staking, NFT Explorer) had its own visual language — built independently by whoever worked on it at the time.

    The result was a product that felt fragmented. Buttons looked different page to page. Data tables had three different row heights. Color usage was inconsistent enough that users couldn't tell which elements were interactive. And every new feature meant re-making decisions that had already been made elsewhere — just forgotten.

    "A design system isn't a deliverable. It's the infrastructure that makes every other deliverable faster and better."

    03 — The challenge

    What was broken

    Problem
    Every feature was designed from scratch — no shared components, no reuse.
    ↓
    Impact
    Designers spent 40–50% of feature time rebuilding UI that already existed elsewhere in the product.
    ↓
    Design implication
    Build a shared component library that covers 90% of recurring UI patterns — so designers solve new problems, not old ones.
    Problem
    No token system meant global visual changes required editing every file manually.
    ↓
    Impact
    When the brand color shifted slightly, it took 2 days to update Figma files. The codebase took longer — and diverged.
    ↓
    Design implication
    Design tokens as the single source of truth — one change in Figma variables propagates everywhere.
    Problem
    Developers implemented UI from Figma by eye — no spec, no handoff standards.
    ↓
    Impact
    Inconsistencies between design and production were constant. QA cycles were long. Designer-developer trust was low.
    ↓
    Design implication
    Tokens mapped directly to CSS variables. Every component ships with documented specs, states, and usage rules.
    04 — Approach

    How I built it

    01
    UI Audit
    Catalogued every screen across all 4 verticals. Documented every button variant, color value, spacing pattern, and typography style in use. Found 14 different button styles, 6 different input heights, and 23 distinct shades of the same grey.
    02
    Token Architecture
    Designed a two-tier token system: primitive tokens (raw values — exact colors, px sizes) and semantic tokens (purpose-driven — "color-action-primary", "spacing-component-padding"). Semantic tokens reference primitives, making rebranding a 5-minute operation.
    03
    Component Prioritization
    Ranked components by frequency of use across the product and by developer implementation cost. Built the highest-impact components first — buttons, inputs, data tables, cards, badges — before moving to complex patterns.
    04
    State Documentation
    Every component was built with all interactive states: default, hover, focus, active, disabled, loading, error. Figma components used auto-layout and boolean properties so engineers could inspect any state directly without asking.
    05
    Dev Handoff
    Worked with the frontend lead to map Figma tokens to CSS custom properties 1:1. Created a shared naming convention so Figma variable names matched CSS variable names exactly — zero translation needed between design and code.
    06
    Team Adoption
    Ran two workshops with the design and engineering team. Created a contribution guide so the system could grow without becoming a bottleneck. Set up a #design-system Slack channel for real-time support during the migration period.
    05 — Principles

    What guided the decisions

    01
    Tokens before components
    Get the token layer right first. A component built on bad tokens is hard to fix at scale. Tokens built right make every component easier to maintain and every future change cheaper.
    02
    Design for the engineer, not just the designer
    The system's real user is the frontend developer implementing it. Every component was designed with implementation in mind — CSS-mappable tokens, predictable state structure, sensible defaults.
    03
    Cover 90%, not 100%
    A system that tries to cover every edge case becomes too rigid to use. I focused on the 90% of UI that recurs — and documented how to handle exceptions — rather than over-engineering for hypothetical futures.
    04
    The system is a product, not a project
    Design systems don't have a launch date — they evolve with the product. I set up contribution workflows, versioning conventions, and a deprecation process from day one so the system could grow without chaos.
    06 — The system

    What was built

    Foundation
    Design Tokens
    ~30 semantic tokens covering color, typography, spacing, radius, shadow, and z-index. Mapped 1:1 to CSS custom properties. A single Figma variable change propagates across the entire product.
    UI Library
    60+ Components
    Buttons, inputs, selects, tables, badges, cards, tooltips, modals, navigation, pagination, charts, and data visualization primitives — all with full state coverage and auto-layout.
    Patterns
    Page Templates
    Reusable page-level patterns for empty states, loading skeletons, error screens, onboarding flows, and settings pages — so new features start from a proven layout, not a blank canvas.
    Documentation
    Inline Specs
    Usage guidelines, do/don't examples, accessibility notes, and handoff specs embedded directly in Figma components. No separate documentation site — context lives where the design lives.
    Accessibility
    WCAG 2.1 AA
    All color combinations verified for contrast. Interactive elements have documented focus states. Error messages meet screen reader standards. Accessibility baked in, not bolted on.
    Theming
    Dark / Light modes
    Semantic token layer makes theme switching a single variable swap. Both dark and light themes fully supported across all components — with DappRadar's dark-first brand as the default.

    08 — Reflection

    What I learned

    The hardest part of building a design system isn't the design — it's the organizational change. Getting engineers to trust the system and designers to contribute to it rather than around it required as much communication as it did craft.
    Token naming is surprisingly consequential. A poorly named token gets used incorrectly — a well-named one enforces intent without documentation. "color-text-muted" is self-explanatory. "grey-500" is not.
    Start with the components that will be challenged first. Building buttons and inputs early meant the most-debated elements were already decided — freeing teams to focus on product logic instead of UI arguments.
    A design system earns trust through consistency of quality, not comprehensiveness. Ship fewer, better components — then expand. A system with 20 perfect components gets adopted faster than one with 100 fragile ones.
